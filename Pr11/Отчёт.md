# Отчёт по второй контрольной работе
В ходе выполнения второй контрольной работы были последовательно реализованы и проанализированы ключевые механизмы аутентификации и авторизации в веб-приложениях.

<h2>Практика 7: Базовые методы аутентификации:</h2>
В рамках данной работы был создан базовый сервер на Node.js с использованием Express, реализующий регистрацию и вход пользователей. Пароли пользователей хранятся в хешированном виде с использованием библиотеки bcrypt, что обеспечивает безопасность учётных данных даже в случае компрометации базы данных.


<h2>Практика 8: JWT-токены:</h2>
На основе предыдущей работы были внедрены JSON Web Tokens для управления сессиями. При успешном входе сервер генерирует JWT-токен, который подписывается секретным ключом и содержит идентификатор пользователя и его базовые данные. Был создан middleware, который проверяет наличие и валидность токена в заголовке Authorization перед предоставлением доступа к защищённым маршрутам. Это позволило разграничить доступ к API и обеспечить безопасную передачу данных между клиентом и сервером без необходимости хранить сессии на сервере.


<h2>Практика 9: Refresh-токены</h2>
Для повышения удобства использования и безопасности была реализована система из двух типов токенов: короткоживущий access-токен (15 минут) для доступа к API и долгоживущий refresh-токен (7 дней) для получения новых пар токенов. Был создан эндпоинт, который принимает refresh-токен, проверяет его валидность и выдаёт новую пару токенов. Это позволяет пользователям оставаться в системе длительное время без повторного ввода пароля, при этом сохраняя высокий уровень безопасности за счёт частой смены access-токенов.

<h2>Практика 10: Клиентская часть с автоматическим обновлением токенов</h2>
На данном этапе был разработан полнофункциональный клиент на React.js, который взаимодействует с сервером через API. В клиентской части реализованы страницы регистрации и входа, а также панель управления товарами с возможностью создания, просмотра, редактирования и удаления товаров. Ключевой особенностью стала реализация перехватчиков в axios, которые автоматически добавляют access-токен в каждый запрос, а при получении ошибки 401 автоматически отправляют запрос на обновление токенов и повторяют исходный запрос.

<h2>Практика 11: реализовано управление доступом на основе ролей (RBAC)</h2>
Финальным этапом стала реализация модели контроля доступа на основе ролей. В систему были введены три роли: пользователь (только просмотр товаров), продавец (создание и редактирование товаров) и администратор (полный доступ, включая управление пользователями и удаление товаров). Был создан middleware, который проверяет роль пользователя из JWT-токена и ограничивает доступ к определённым эндпоинтам.
